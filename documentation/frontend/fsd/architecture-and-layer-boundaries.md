# Feature-Sliced Design (FSD) — 철학과 레이어 경계

이 문서는 Factory MES 프론트엔드에서 FSD를 적용·개편할 때 참고하는 **아키텍처 기본 철학**과, 특히 **widgets vs features** 경계를 팀에서 어떻게 정할지에 대한 **판단 기준·근거**를 정리한다.

---

## 1. FSD의 기본 철학

### 1.1 목표

- **비즈니스 도메인**을 기준으로 코드를 나누어, 한 영역의 변경이 다른 영역에 불필요하게 퍼지지 않게 한다.
- **의존 방향을 한 방향으로 고정**하여 “아래층이 위층을 알지 못하게” 하여 결합도를 낮춘다.
- **공개 API(public API)**로 슬라이스 간 접점을 제한해, 내부 구현 변경이 외부에 미치는 영향을 줄인다.

### 1.2 레이어(위에서 아래로)

전통적인 FSD에서는 대략 다음 순서의 **층**을 둔다. 위층은 아래층만 사용할 수 있다.

| 레이어 (상→하) | 역할 (요약) |
|----------------|-------------|
| **app** | 라우팅, 전역 프로바이더, 진입점, 앱 단위 설정 |
| **pages** | 라우트 단위 화면; 위젯·피처를 조합해 “이 URL에서 보이는 것”을 완성 |
| **widgets** | 화면의 **구역(block)** — 여러 feature·entity·shared를 **조합**한 복합 UI |
| **features** | 사용자 **시나리오·행동** 한 덩어리 (한 가지 “할 수 있는 일”) |
| **entities** | 비즈니스 엔티티(Asset, Relationship 등) — UI·API·모델의 도메인 단위 묶음 |
| **shared** | 도메인에 묶이지 않는 공용 UI·유틸·API 클라이언트 등 |

프로젝트 규모에 따라 **processes** 층을 두기도 하고, 초기에는 **widgets / features**를 아예 생략하고 `pages` + `entities` + `shared`만 쓰기도 한다. 중요한 것은 **의존이 위→아래로만 흐른다**는 규칙이다.

### 1.3 슬라이스와 세그먼트

- **슬라이스**: 레이어 안에서 도메인·기능 단위 폴더 (예: `entities/asset`, `features/create-relationship`).
- **세그먼트**: 슬라이스 내부의 `ui`, `model`, `api`, `lib` 등 역할별 하위 구조.

외부에서는 슬라이스 루트의 **index(공개 API)**만 import 하는 것을 목표로 한다.

### 1.4 왜 “행동(features)”과 “조합(widgets)”을 나누는가

- **feature**는 “사용자가 제품으로 **무엇을 하는지**”에 가깝다. 테스트·요구사항·API 플로우와 정렬하기 쉽다.
- **widget**은 “화면 **어느 구역**에 무엇을 **함께** 보여줄지”에 가깝다. 같은 구역을 여러 페이지에서 재사용할 때 유리하다.

둘을 혼동하면 “큰 컴포넌트 하나”에 시나리오·레이아웃·여러 API가 섞여, 재사용과 변경 범위 파악이 어려워진다.

---

## 2. 의존 방향 (widgets ↔ features)

FSD에서 일반적으로:

- **widgets → features → entities → shared** 순으로만 의존한다.
- **feature가 widget을 import 하면 안 된다.** (아래층이 위층을 바라보는 형태가 되기 때문)

따라트:

- **feature**: 특정 시나리오 구현에 집중. 가능하면 한·소수 entity와 맞닿게 유지.
- **widget**: 여러 feature를 **배치·연결**하는 역할. 비즈니스 규칙의 “본체”는 feature·entity에 두는 편이 유지보수에 유리하다.

---

## 3. widgets vs features — 판단 기준 (체크리스트)

팀에서 경계를 정할 때 아래를 **순서대로** 적용하면 수렴이 빠르다.

1. **동사 하나로 이름을 짓기 쉬운가?**  
   예: “관계를 만든다”, “시뮬레이션을 실행한다”, “ObjectType 스키마를 저장한다” → **feature** 후보가 강하다.  
   “왼쪽 사이드 전체”, “캔버스 상단 툴바 영역”처럼 **구역** 이름이 자연스러우면 **widget** 후보다.

2. **다른 페이지에서 같은 UI 덩어리를 그대로(또는 거의 그대로) 쓸 가능성이 있는가?**  
   있으면 바깥 **조합**은 **widget**, 그 안의 실제 행동은 **feature**로 쪼개는 패턴이 잘 맞는다.

3. **한 덩어리가 서로 다른 “행동”을 여러 개 노출하는가?**  
   예: 사이드 패널에 스키마 편집·관계 생성·시뮬 제어가 함께 있으면  
   - 탭/섹션 **배치** → **widget**  
   - 각 탭의 폼·API·검증 → 각각 **feature** (또는 하나의 feature 폴더 안의 여러 UI 모듈)

4. **테스트·스토리 단위가 무엇인가?**  
   - “이 시나리오가 API와 함께 동작한다” → **feature** 경계에 맞춘다.  
   - “여러 feature가 한 구역에 배치됐을 때 레이아웃·가시성이 맞다” → **widget** 테스트.

5. **의존하는 엔티티·API의 개수**  
   한 entity·API에 깊게 묶인 로직 → **feature** 쪽에 두기 쉽다.  
   여러 entity API와 여러 폼을 한 구역에서 엮음 → 바깥은 **widget**, 안쪽 로직은 **feature**로 나누는 것이 안전하다.

**한 줄 휴리스틱:**  
- “**무엇을 한다** (use case)” → **feature**  
- “**어디에 무엇들을 모아 보여준다** (composition)” → **widget**

## 4. 팀이 선택할 수 있는 적용 스펙트럼

규모와 재사용 계획에 따라 아래 중 하나를 택하거나 혼합할 수 있다.

| 옵션 | 설명 | 적합한 상황 |
|------|------|-------------|
| **A. 엄격 FSD** | pages는 얇게; 큰 구역은 widget, 행동은 feature로 분리 | 인원·페이지가 늘고, 단일 페이지(예: 캔버스)가 계속 비대해질 때 |
| **B. features만 추가** | widget 층 없이 feature + entities로만 분해; pages는 다소 두꺼워도 허용 | 팀이 작고, 동일 블록을 다른 라우트에 붙이는 일이 거의 없을 때 |
| **C. widget은 얇게** | 실제로 2개 이상 페이지에서 재사용되는 조합만 widget; 나머지는 `pages/.../ui`에서 파일 분할만 | widget 남발을 피하고 싶을 때. 실질적으로 B에 가깝게 시작하는 경우가 많음 |
| **D. 페이지 단위 bounded context** | 예: `pages/canvas` 아래에 전용 `widgets`/`features` 폴더로만 구조화 | 전역 FSD 규칙을 한 번에 맞추기 어려울 때, 한 화면부터 정리 |

작은 팀에서는 **C 또는 D로 시작**하고, 두 번째 페이지에서 같은 블록이 필요해지면 그때 **widget으로 승격**하는 방식이 부담이 적다.

---

## 5. 흔한 실수와 대안

- **“크기가 크면 widget”만으로 분리** → 여러 시나리오가 한 덩어리에 섞인다. **책임(행동 vs 조합)**이 우선이어야 한다.
- **feature끼리 과도하게 서로 import** → 공통 부분은 **entity** 또는 **shared/lib**로 내리고 feature는 얇게 유지한다.
- **widget에 API 호출·도메인 규칙을 전부 집중** → 이후 feature로 다시 쪼개야 하는 비용이 크다. widget은 **배치·props 연결**에 가깝게 두는 것이 이상적이다.

---

## 6. 팀 합의문 예시 (한 줄 규칙)

문서나 PR 템플릿에 다음 중 팀이 동의한 문장을 적어 두면 리뷰 시 논쟁이 줄어든다.

- 사용자 시나리오 단위(한 API 플로우 또는 한 폼 제출 단위)는 **feature**에 둔다.
- 두 개 이상의 feature를 한 구역에서 묶어 **다른 라우트에서도 재사용**할 때만 **widget**을 만든다.
- 재사용 계획이 없으면 **`pages/.../ui`에 파일만 분리**하고, 동일 블록이 두 페이지에 보이면 그때 widget으로 승격한다.

---

## 7. 본 저장소와의 연결

- 루트의 `servers/frontend`는 문서상 FSD를 따르며, 현재는 주로 **app / pages / entities / shared**만 명시된 상태일 수 있다.  
- **캔버스 페이지**(`pages/canvas`)처럼 한 파일에 패널·모달·헬퍼가 몰린 경우, 위 기준에 따라 **feature·widget 층 도입** 또는 **pages 내 파일 분할**로 단계적 개편을 검토한다.
- 세부 화면 설명은 `documentation/frontend/`의 기존 문서(캔버스, entities 등)를 참고한다.
